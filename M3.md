# Crooshubext — метрики для Grafana

**Сервис:** Crooshubext — маршрутизатор запросов к AI-агентам.

Принимает входящие вызовы по HTTP, определяет нужного агента и доставляет запрос либо по REST, либо через Kafka. Возвращает ответ агента вызывающей стороне.

**Формат метрик:** Prometheus. Неймспейс — `crooshubext`.

**Версия приложения** передаётся через label `version` во всех метриках сервиса (см. ниже).

---

## Версия приложения

Отдельная метрика-маркер, которая позволяет видеть в Grafana какая версия сейчас запущена и когда происходили деплои.

### `crooshubext_build_info`
**Тип:** Gauge (всегда равен `1`)
**Описание:** Информация о запущенном инстансе сервиса. Не несёт числового смысла — используется только для получения labels через `info`-join или отображения в панели.

| Label | Значения | Описание |
|---|---|---|
| `version` | `1.2.3` | Семантическая версия сборки |
| `commit` | `a3f1c9d` | Короткий git-хэш коммита |
| `build_time` | `2026-05-28T10:00:00Z` | Время сборки образа |

**Панель:** Stat-панель с текущей версией:
```
crooshubext_build_info
```
Grafana отобразит label `version` как текст. При деплое новой версии на графиках появится вертикальная аннотация — старое значение исчезнет, новое появится.

**Как добавить версию к другим метрикам** (info-join в PromQL):
```
rate(crooshubext_http_requests_total[5m])
* on() group_left(version)
crooshubext_build_info
```
Это позволяет разбивать любой график по `version` без добавления label в каждую метрику отдельно.

## Входящие запросы (Inbound HTTP)

Сервис принимает запросы на `/api/v1/invoke`. В будущем эндпоинты будут добавляться — все они попадают в одну метрику, различаясь по label `endpoint`.

---

### `crooshubext_http_requests_total`
**Тип:** Counter
**Описание:** Общее количество входящих HTTP-запросов к сервису.

| Label | Значения | Описание |
|---|---|---|
| `method` | `POST`, `GET`, ... | HTTP-метод |
| `endpoint` | `/api/v1/invoke` | Шаблон пути. Переменные части URL (ID, хэши) должны быть заменены на шаблон, например `/api/v1/invoke/{id}` — иначе каждый уникальный URL создаст отдельный time series |
| `status_code` | `200`, `400`, `500`, ... | HTTP-код ответа |
| `agent_id` | `agent-abc`, `agent-xyz`, ... | К какому агенту направлен запрос |
| `routing_mode` | `rest`, `kafka` | Способ доставки запроса агенту |

**RPS по эндпоинтам:**
```
sum by (endpoint, method) (
  rate(crooshubext_http_requests_total[1m])
)
```

**Error rate (%) по эндпоинтам:**
```
100 * sum by (endpoint) (
    rate(crooshubext_http_requests_total{status_code=~"5.."}[5m])
  )
  / sum by (endpoint) (
    rate(crooshubext_http_requests_total[5m])
  )
```

**RPS разбивка по агентам:**
```
sum by (agent_id, routing_mode) (
  rate(crooshubext_http_requests_total[1m])
)
```

---

### `crooshubext_http_request_duration_seconds`
**Тип:** Histogram
**Описание:** Полное время обработки входящего запроса — от получения до отправки ответа клиенту. Включает время ожидания ответа агента.

| Label | Значения | Описание |
|---|---|---|
| `method` | `POST`, `GET`, ... | HTTP-метод |
| `endpoint` | `/api/v1/invoke` | Шаблон пути |
| `status_code` | `200`, `500`, ... | Код ответа |
| `agent_id` | `agent-abc`, `agent-xyz`, ... | К какому агенту шёл запрос |

**Buckets:** `0.1, 0.25, 0.5, 1, 2, 5, 10, 30, 60` секунд (AI-агенты могут отвечать долго).

**p99 latency по эндпоинтам:**
```
histogram_quantile(0.99,
  sum by (endpoint, le) (
    rate(crooshubext_http_request_duration_seconds_bucket[5m])
  )
)
```

**p99 latency по агентам:**
```
histogram_quantile(0.99,
  sum by (agent_id, le) (
    rate(crooshubext_http_request_duration_seconds_bucket[5m])
  )
)
```

---

### `crooshubext_http_requests_in_flight`
**Тип:** Gauge
**Описание:** Сколько входящих запросов сейчас находится в обработке (ожидают ответа агента).

| Label | Значения | Описание |
|---|---|---|
| `endpoint` | `/api/v1/invoke` | Шаблон пути |
| `agent_id` | `agent-abc`, `agent-xyz`, ... | К какому агенту ждут ответа |

**Текущая нагрузка:**
```
sum by (agent_id) (crooshubext_http_requests_in_flight)
```

## Исходящие вызовы агентов (Outbound)

Сервис вызывает агентов двумя способами: синхронно по REST или асинхронно через Kafka. Метрики описываются отдельно для каждого способа.

---

### REST-вызовы агентов

#### `crooshubext_agent_http_requests_total`
**Тип:** Counter
**Описание:** Количество исходящих HTTP-запросов к агентам.

| Label | Значения | Описание |
|---|---|---|
| `agent_id` | `agent-abc`, `agent-xyz`, ... | Идентификатор агента |
| `method` | `POST`, `GET`, ... | HTTP-метод |
| `endpoint` | `/invoke`, `/run`, ... | Путь на стороне агента (шаблон) |
| `status_code` | `200`, `400`, `500`, `503`, ... | HTTP-код ответа агента |

**Error rate по каждому агенту:**
```
100 * sum by (agent_id) (
    rate(crooshubext_agent_http_requests_total{status_code=~"5.."}[5m])
  )
  / sum by (agent_id) (
    rate(crooshubext_agent_http_requests_total[5m])
  )
```

**RPS по агентам:**
```
sum by (agent_id) (
  rate(crooshubext_agent_http_requests_total[1m])
)
```

---

#### `crooshubext_agent_http_request_duration_seconds`
**Тип:** Histogram
**Описание:** Время ответа агента (от отправки запроса до получения ответа, включая сеть).

| Label | Значения | Описание |
|---|---|---|
| `agent_id` | `agent-abc`, `agent-xyz`, ... | Идентификатор агента |
| `status_code` | `200`, `500`, ... | Код ответа |

**Buckets:** `0.1, 0.25, 0.5, 1, 2, 5, 10, 30, 60` секунд.

**p99 latency по каждому агенту:**
```
histogram_quantile(0.99,
  sum by (agent_id, le) (
    rate(crooshubext_agent_http_request_duration_seconds_bucket[5m])
  )
)
```

---

### Kafka — отправка задач агентам (Producer → `topic-in`)

Когда `routing_mode = kafka`, сервис публикует задачу в `topic-in` и ожидает ответа из `topic-out`.

#### `crooshubext_kafka_producer_messages_total`
**Тип:** Counter
**Описание:** Количество сообщений, отправленных в `topic-in`.

| Label | Значения | Описание |
|---|---|---|
| `topic` | `topic-in` | Топик назначения |
| `agent_id` | `agent-abc`, `agent-xyz`, ... | Для какого агента задача |
| `status` | `success`, `error` | Удалось ли отправить |

**Throughput:**
```
sum by (agent_id) (
  rate(crooshubext_kafka_producer_messages_total{status="success"}[1m])
)
```

**Error rate продюсера:**
```
100 * sum(rate(crooshubext_kafka_producer_messages_total{status="error"}[5m]))
    / sum(rate(crooshubext_kafka_producer_messages_total[5m]))
```

---

#### `crooshubext_kafka_producer_duration_seconds`
**Тип:** Histogram
**Описание:** Время от вызова produce до подтверждения от брокера (ack).

| Label | Значения | Описание |
|---|---|---|
| `topic` | `topic-in` | Топик |

**Buckets:** `0.001, 0.005, 0.01, 0.05, 0.1, 0.5, 1` секунд.

**p99 задержки публикации** (если растёт — проблема на стороне брокера):
```
histogram_quantile(0.99,
  sum by (topic, le) (
    rate(crooshubext_kafka_producer_duration_seconds_bucket[5m])
  )
)
```

---

### Kafka — получение ответов агентов (Consumer ← `topic-out`)

#### `crooshubext_kafka_consumer_messages_total`
**Тип:** Counter
**Описание:** Количество сообщений (ответов агентов), полученных из `topic-out`.

| Label | Значения | Описание |
|---|---|---|
| `topic` | `topic-out` | Топик |
| `agent_id` | `agent-abc`, `agent-xyz`, ... | От какого агента ответ |
| `status` | `success`, `error` | Успешно ли обработан ответ |

**Rate обработки ответов:**
```
sum by (agent_id) (
  rate(crooshubext_kafka_consumer_messages_total[1m])
)
```

**Error rate консюмера:**
```
100 * sum(rate(crooshubext_kafka_consumer_messages_total{status="error"}[5m]))
    / sum(rate(crooshubext_kafka_consumer_messages_total[5m]))
```

---

#### `crooshubext_kafka_consumer_lag`
**Тип:** Gauge
**Описание:** Отставание консюмера — разница между последним offset в партиции и текущим прочитанным. Рост лага означает, что агенты отвечают быстрее, чем сервис успевает забирать ответы.

| Label | Значения | Описание |
|---|---|---|
| `topic` | `topic-out` | Топик |
| `partition` | `0`, `1`, ... | Номер партиции |
| `consumer_group` | `crooshubext-cg` | Группа консюмера |

> Рекомендуется собирать через `kafka-exporter` на уровне инфраструктуры, а не из приложения — уточнить с DevOps.

**Суммарный лаг по топику:**
```
sum by (topic) (crooshubext_kafka_consumer_lag)
```

**Скорость изменения лага** (положительный — лаг растёт, отрицательный — нагоняет):
```
deriv(sum by (topic) (crooshubext_kafka_consumer_lag)[5m:])
```

## Рекомендуемые панели дашборда

| Панель | Метрика | Формула |
|---|---|---|
| Версия сервиса | `build_info` | `crooshubext_build_info` |
| RPS входящих запросов | `http_requests_total` | `sum by (endpoint) (rate(...[1m]))` |
| Error rate входящих | `http_requests_total` | `100 * rate({status_code=~"5.."}[5m]) / rate([5m])` |
| Latency p99 по эндпоинтам | `http_request_duration_seconds` | `histogram_quantile(0.99, ...)` |
| In-flight запросы | `http_requests_in_flight` | `sum by (agent_id) (...)` |
| Error rate по агентам (REST) | `agent_http_requests_total` | `100 * rate({status_code=~"5.."}[5m]) / rate([5m])` |
| Latency агентов (REST) | `agent_http_request_duration_seconds` | `histogram_quantile(0.99, ...)` |
| Kafka producer throughput | `kafka_producer_messages_total` | `sum by (agent_id) (rate({status="success"}[1m]))` |
| Kafka consumer lag | `kafka_consumer_lag` | `sum by (topic) (...)` |
| Kafka consumer rate | `kafka_consumer_messages_total` | `sum by (agent_id) (rate(...[1m]))` |

## Алерты

| Алерт | Условие | Severity |
|---|---|---|
| Высокий error rate на входе | error rate > 5% за 5 мин | critical |
| Деградация latency входящих | p99 > 30s за 5 мин | warning |
| REST-агент недоступен | error rate агента > 10% за 5 мин | critical |
| Kafka producer не пишет | rate = 0 при наличии in-flight kafka-запросов | critical |
| Kafka consumer lag растёт | `deriv(lag[5m]) > 0` стабильно 10 мин | warning |
| Kafka consumer lag критичный | lag > 2000 | critical |
